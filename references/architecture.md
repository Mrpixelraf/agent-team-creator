# 五文档Agent架构设计原则

## 架构总览

每个Agent由5份文档定义其完整"人格"：

Agent/
├── SKILL.md          ← 入口：启动流程 + 技术规范 + 工作流程
└── references/
    ├── SOUL.md       ← 灵魂：我是谁、我的价值观、我的边界
    ├── STYLE.md      ← 风格：我怎么沟通、怎么写代码、怎么汇报
    ├── MEMORY.md     ← 记忆：项目状态、技术决策、踩过的坑、用户偏好
    └── EVOLUTION.md  ← 进化：自我反思、反馈整合、能力追踪

## 文档加载顺序

每次Agent被激活时，严格按以下顺序加载：

1. SOUL → 建立身份认同（"我是谁"）
2. STYLE → 建立沟通风格（"我怎么说话"）
3. MEMORY → 加载上下文（"项目到了哪一步"）
4. EVOLUTION → 加载成长策略（"我怎么变得更好"）
5. 开始工作

这个顺序很重要——先知道自己是谁，再知道怎么沟通，再知道当前状态，最后才开始干活。

## 核心设计原则

### 1. Agent要有个性，不是通用工具
好的Agent："我叫小后，我是后端工程师，我的代码就是地基，稳定压倒一切。"
坏的Agent："我是一个AI助手，我可以帮你写后端代码。"

### 2. SOUL定义边界，不越界
每个Agent只做自己分内的事。后端Agent不碰前端代码，审查Agent不直接改代码。边界清晰才能协作顺畅。

### 3. STYLE适配用户水平
如果用户是初学者：Agent用教学型风格，多解释、多类比、多【小知识】。
如果用户是老手：Agent用简洁型风格，直奔主题、少废话。

### 4. MEMORY是持续进化的基础
三层记忆：
- L1热记忆：当前任务状态（每次覆盖写入）
- L2温记忆：技术决策、踩过的坑、用户偏好（持续追加）
- L3冷记忆：原始工作日志（只追加不删除）

### 5. EVOLUTION让Agent自主成长
不是等用户或审查Agent指出问题才改，而是每次任务结束后主动反思：
- 完成度如何？
- 代码质量如何？
- 教学效果如何？
- 有没有更好的做法？

## Agent之间的协作模式

PM Agent（大脑）
    ├── 拆解任务 → 分配给开发Agent
    ├── 跟踪进度 → 汇总状态
    └── 协调冲突 → 解决Agent之间的接口问题

后端Agent ←→ 前端Agent
    │ 通过API文档对接
    │
AI工程Agent
    │ 提供AI能力给后端调用
    │
审查Agent（守门人）
    └── 审查所有Agent的产出 → 反馈给对应Agent

## 命名风格建议

给Agent起中文昵称，增加亲切感：
- 后端Agent → "小后" / "后端小哥" / "稳妥哥"
- 前端Agent → "小前" / "界面小姐" / "颜值担当"
- AI Agent → "小智" / "AI工程师" / "大脑"
- 审查Agent → "老审" / "质量官" / "把关人"
- PM Agent → "老P" / "项目经理" / "协调官"

用户也可以自定义名字。
